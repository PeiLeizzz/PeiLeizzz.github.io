# 第五章 系统分析

概念：理解和发现用户的要求，也称为需求分析

任务：识别需求和表达需求，调查和分析业务流程，建立系统的逻辑模型

系统调查方法：资料收集、访谈、实地观察、问卷调查

需求引导方法：

  • 原型法：利用快速开发工具，根据用户的初步需求，构造出信息系统的初步原型

    能准确的反应用户的需求，但需要额外的工作量

  • 联合应用开发（JAD）会议

  • 观摩法

# 需求分析

类型：业务性需求和技术性需求

方法：

  • 面向过程的结构化方法（自顶向下、逐层分解）
  • 面向数据的信息工程方法（数据驱动）
  • 面向对象方法（对象驱动、UML）

# 系统分析建模

方法：

  • 流程建模
    • 业务流程（业务流程图、UML 活动图）
    • 数据处理流程（数据流图）
  • 用例建模
    • 信息系统功能建模（UML 用例图）
  • 领域对象建模
    • 领域对象模型（UML 类图、UML 状态图）

# 第七章 用例建模

# 基于用例的需求分析

用例分析是站在最终用户的角度看待系统及其特性，结合面向对象方法

用例：是对于一组动作序列的描述，系统执行这些动作会对特定的参与者产生可观测的、有价值的结果。用于描述功能性的需求，每个用例至少和一个参与者相关。

参与者：系统之外与系统进行直接交互的任何事物（使用系统的个人、系统所连接的外部硬件、与该系统进行通信的其他信息系统）

外部实体：数据的来源和去向，提供数据的人员不一定会执行系统的功能

参与者的泛化：抽取一些角色之间的共性，作为父角色

业务用例：具体业务相关，如“上门维修”

系统用例：有信息处理的那部分功能，如“填写派工单”

# 用例的描述

用例规格说明：

  • 用例名称
  • 主要参与者 / 次要参与者
  • 简要描述
  • 前置条件：如预定图书前需要登录
  • 后置条件：如预定图书后图书状态改变
  • 主事件流
  • 备选事件流
  • 特殊要求 / 非功能性需求
  • 发生频率
  • 业务规则

UML 顺序图：更直观地表现出参与者和系统之间的交互

# 用例之间的关系

拓展和包含关系就是用例模型中消除冗余的一种手段。

用例关系：

  • 包含关系：<<include>> 虚线，主用例指向子用例

  • 拓展关系：<<extend>> 虚线，子用例指向主用例

  • 泛化关系：父用例、子用例（子用例是相互独立而完整的）

# 第八章 领域对象建模

领域对象建模解决业务管理对象到信息系统逻辑结构（软件结构模型和数据库结构模型）的映射。

# 面向对象方法

对象：一些属性及专用服务的封装体，是问题域中一些事物的抽象

属性：类的特征或特性

方法：对象的行为

类：具有相同属性和服务的一组对象的抽象概念

实例：一个具体的对象

类是静态的，对象是动态的

抽象类：表征在对问题领域进行分析、设计的抽象概念

接口：抽象类的变体，是一些方法的集合，但所有的方法都是抽象的

封装：信息隐藏，保证软件部件具有较好的模块性（黑箱）。对象是一个更高级别的封装体，将数据和服务封装于一个内在的整体，向外提供接口，隐藏内部实现

消息：对象发出的服务请求（对象间的交互信息）

继承：特殊类的对象拥有其一般类的全部属性与服务，这个过程称为泛化

多态：相同的操作可作用于多种类型的对象并获得不同的结果（建立在继承机制之上）

  • 覆盖：子类对父类的方法进行重写，运行时多态、动态多态
  • 重载：一个类中定义多个名称相同、参数类型不同的方法，编译时多态、静态多态

类之间的联系方式：

  • 继承 / 泛化
  • 实现

对象关系：

  • 关联：静态联系，长期联系，比如整体和部分关系（对象引用）
    • 聚合：部分可以脱离于整体，有各自的生命周期（比如公司和员工）
    • 组合：部分不能脱离整体，生命周期一致（比如人和人的大脑）
  • 依赖：动态联系，运行时联系，短期联系（函数中引用、函数返回值）

耦合性：依赖 < 关联 < 聚合 < 组合 < 泛化

重用：对象具有封装性和信息隐蔽等特性，容易实现软件重用

面向对象方法的优势：

  • 更接近人类的思维方式
  • 生命周期各阶段过渡平滑
  • 维护更加容易
  • 更加稳定
  • 与数据模型一致（ER 图中的实体）

# 识别领域对象

采用的分析技术：

  • 类图：对象及其关系，描述系统的静态结构
  • 状态图：对象的状态及转换，用于描述基于事件响应的对象动态行为和状态之间的关系

领域对象：问题域中有意义的概念类，是现实系统中的事物或事件

# 对象的关联

关联表示不同类的对象之间的静态关系，它在一段时间内将多个类的实例连接在一起。（has a）

描述关联的要素：

  • 关联名称：二元，反映关系的目的
  • 对象在关联中的角色
  • 多重性：一对一、一对多、多对多
  • 导向性：单向访问，通常是源对象存储了对目标对象的一些引用

整体-部分关联（a is a part of b）：

  • 对象 a 是对象 b 的一个组成部分，描述组装关系，如机器和零件、班级和学生）

  • 也称为聚集，UML 中使用连接线和菱形表达，菱形一端的对象是整体对象

    • 组合聚集（组成）：实心菱形，具有很强的归属关系，在特定时刻部分只能是一个组合对象的成员，整体端的重数不会超过 1（部分无法被多个整体对象共享）

    • 共享聚集（聚集）：空心菱形，部分可能同时属于多个整体对象

# 对象的泛化

泛化是在多个概念之间识别共性，定义超类和子类关系的活动

一般-特殊结构（is a kind of）：类 A 具有类 B 的全部属性和行为，而且具有自己特有的某些属性和服务

不要将一个对象的状态变化设计为多个子类，除非对象的多数行为是由状态来决定

# 第九章 系统设计概述

系统设计：逻辑模型 -> 实际系统；针对需求，进行计算机信息系统的物理模型设计（怎么做）

一个逻辑模型，可以提出多个物理模型

基本设计方法：模块化、层次化、封装、契约

系统设计的内容：

  • 总体设计：设计软件的体系结构（架构）、软件结构、系统对外接口和服务
  • 详细设计：具体细节

软件结构

  • 基于模块封装

    • 自顶向下、逐层分解（结构化设计）
    • 主要模型：模块结构图（功能结构图）

  • 基于对象封装

    • 强调面向对象

    • 主要模型：类图

      类的方法本质上也是模块封装

  • 基于服务封装：

    • 面向服务的架构（SOA）：操作、服务、业务流程
    • 主要模型：构件图（底层基于类来实现）

软件设计的两类模型

  • 静态模型
    • 主要表示软件结构，即组成元素及其关系。一种是开发态的源程序结构，另一种是运行态的构件结构。
    • 模块结构图（结构化方法）、类图（面向对象方法）、构件图（面向服务的架构）
  • 动态模型
    • 主要表示软件执行动作的步骤和流程控制
    • 程序结构图（结构化方法）、顺序图（面向对象方法）

# 第十章 系统总体设计

# 软件架构的设计

架构：系统最高层的设计方案。架构包含系统的一组基本结构，每种结构都有各种类型的部件及其关系构成，架构描述了这些部件的组合、相互调用参照、通信以及其他动态交互。

软件架构：一个应用程序或计算机系统的软件架构是一个或一组结构，它包含组成系统的软件元素、这些元素对外可见的性质以及它们之间的关系。对外可见的性质指软件元素能够提供的服务、性能特征、错误处理、共享资源的用法等。

软件架构模式：可重复使用的软件结构风格

# 多层应用架构模式

分层：将整个任务划分为不同级别（横向、纵向）

三层架构：UI（表示层）、BLL（业务逻辑层）、DAL（数据库访问层）

MVC 架构：View -> Controller -> Model

五层架构：表现层、控制层/中介层、领域层、数据映射层、数据访问层

SSM 五层：View -> Controller -> Service -> Entity -> DAO

多层体系结构的优势：安全性提高、可靠性和响应速度提高、重用性和可维护性提高、并行开发能力提高

# 软件框架

软件框架：对整个或部分系统可重用的设计和实现

SSH: Struts, Spring, Hibernate

SSM: Spring, SpringMVC, Mybatis

# 高层结构设计

高层结构讨论系统比较大的组成部件（包、构件、子系统等）及其接口设计

包：逻辑分组的手段，将相关成分聚在一起，以构成更高层的组织单元。最常用的方法是将类以包为单位进行分组，一个包可以包含其他的包。

在 Java 中，包还指代了物理的组织手段

分包原则：

  • 共同封闭原则：一个包中的各个类应该是由于相似的原则而改变，即将一组指责相似、但以不同方式实现的类归为一个包中。
  • 共同复用原则：一个包中的各个类应该一起被复用，复用其中一个可能需要同时考虑同一个包中的其他协作类，通常和业务功能相关。

包图：描述包及其依赖关系

子系统：当按照相对完整和独立的业务功能或管理职能组织包，并对这样的包进行封装后，一个高层的、具有特定功能的、可以运行的独立构件就产生了，称为子系统。子系统对外可以提供有限的接口。

UML 构件图：

接口：两个不同系统（或子程序）交接并通过它彼此作用的部分。

子系统和包的差异：

  • 子系统是可以直接运行的，具有物理意义
  • 包不提供行为，只是一种容纳各种其他模型元素的容器，具有逻辑意义

子系统的关系：

  • 数据接口：定义可以理解的数据格式和数据文件，子系统之间采用数据文件进行通信
  • 服务接口：定义可以理解的软件服务，子系统之间采用发送服务请求和响应进行通信

构件：系统中实际存在的可更换部分，实现了特定的功能，符合一套接口标准并实现一组接口。构件是可复用的软件组成成分，可被用来构造其他软件。

子系统和构件的差异：

  • 子系统是一个系统，是具有特定功能的整体，可以直接使用，不会被复用
  • 构件只是构成系统的元素，不具备单独使用的能力，用于复用

# 面向对象设计方法

三层分层模式：

  • 分析模型的领域类导出设计阶段中的实体类

  • 增加边界类和控制类完成程序的交互和控制

  • UML 构造型

  • Rose 图符

边界类：完成系统与其参与者之间的交互（接收来自用户和外部系统的信息与请求；将信息与请求提交给用户和外部系统）

每个边界类至少应该与一个参与者相关，根据用例图，每个参与者与一个用例交互，必定导出一个边界类。

实体类：来源于领域模型的类，是一个软件对象，表达了领域对象的信息，以及具有与它所表示的信息有关的操作。

实体类反映的信息需要在系统中进行处理，并需要进行持久化存储。

边界类仅负责数据的输入和输出，不承担业务逻辑；通过与实体类的交互，获得有关数据处理的结果

控制类：控制类代表协调、排序、事务处理以及对其他对象的控制，经常用于封装与某个具体用例有关的控制流。

控制类不与参与者交互，也不封装长效持久的信息

职责：类的方法是对象应该执行的操作，也称为对象的职责和义务。

职责类型：行为型（对象本身的方法）、了解型（对象应掌握的信息，set/get 方法，关联对象）

职责分配方法：

  • 交互图：将消息映射为类的操作，并在实现时转化为类的方法

    交互图类型：顺序图、交互概览图、通信图、计时图

  • CRC（类-职责-协作）卡片法

顺序图：

  • 元素：参与者、对象、生命线、激活框、消息、控制框架（分支、循环、可选）

  • 参与者、对象表示方法

  • 消息：发送消息——实线，响应消息——虚线

  • 控制流：loop alt 等

    复杂控制逻辑由活动图描述更清晰，但活动图不能标识消息

协作图：与顺序图等价

消息的设计：

  • return := message(parameter: parameterType): returnType

    如 reader := getReader(cardID: String)

特殊的消息：对象创建、属性存取

设计类的关系：

  • 泛化（类）——继承、多态

  • 关联（对象）

    导航性：单向关联、双向关联

    关联重数：一对一、一对多、多对多

  • 实现（类）

  • 依赖（对象）——一个类的对象实例使用了另一个类的属性和方法

# 设计原则

  • 总的原则

    • 抽象和复用（封装、信息隐藏）
    • 松耦合

  • 面向功能模块

    • 避免使用全局数据
    • 参数传递尽量少
    • 模块内语句一般为 50-100 行

  • 面向对象

    • 单一职责：SRP，即内聚性原则

    • 开放封闭：OCP

      软件实体（类、模块、函数）应该是可扩展的，又是不可修改的

      即对于扩展开放，对于更改封闭

    • 里氏替换：LSP

      子类型必须能够替换掉它们的基类型

      是多态顺利实现的保证：增加或修改任何一个子类型，基类型不用修改；基类的使用者通过多态得到扩展或修改过的行为

    • 依赖倒置：DIP

      高层模块不应该依赖于底层模块，二者都应该依赖于抽象

      抽象不应该依赖于细节，细节应该依赖于抽象

      让高层需要的服务声明为抽象接口，高层使用这些接口，底层模块实现这些接口

  • 面向服务

# 第十一章 系统详细设计

# 人机交互设计

目标：

  • 降低学习时间
  • 提高执行速度
  • 减少用户出错率
  • 提高记忆保持能力
  • 提高主观满意度

图形用户界面（GUI）是人机交互的主流
