# 软件工程概述

# 软件工程

  • 概念

    • 软件工程是指应用计算机科学、数学及管理科学等原理，以工程化的原则和方法来解决软件问题的工程

  • 目的

    • 提高软件生产率
    • 提高软件质量
    • 降低软件成本

  • 软件工程学

    • 软件开发技术

      • 软件开发方法学
      • 软件工具
      • 软件工程环境

    • 软件工程管理

      • 软件工程管理学
      • 软件经济学

# 计算机软件

  • 计算机软件是指计算机系统中的程序及其文档

  • 程序

    • 程序是计算任务的处理对象和处理规则的描述

    • 计算任务

      • 任何以计算机为处理工具的任务都是计算任务

    • 处理对象

      • 数据
      • 信息

    • 处理规则

      • 处理的动作和步骤

    • 文档

      • 便于了解程序所需的阐述性资料

  • 分类

    • 系统软件

      • 一整套服务于其他程序的程序

      • 特点

        • 与硬件大量交互
        • 多用户大量使用
        • 需要调度、资源共享和复杂进程管理的同步操作
        • 复杂的数据结构
        • 多种外部接口

    • 应用软件

      • 解决特定业务需要的独立应用程序

    • 工程/科学软件

      • 数值计算
      • 计算机辅助设计
      • 系统仿真

    • 嵌入式软件

      • 存在于某个产品或系统中

    • 产品线软件

      • 为多个不同用户的使用提供特定功能
      • 特定的专业市场
      • 大众消费品市场

    • Web 应用

      • 以网络为中心的软件

      • 特点

        • 具备网络密集性
        • 并发性
        • 无法预知的负载量
        • 性能
        • 可用性
        • 数据驱动属性

    • 人工智能软件

    • 开放计算

    • 网络资源

    • 开源软件

# 软件工程基本原理

  • 是确保软件产品质量和开发效率的原理的最小集合

  • 用分阶段的生命周期计划严格管理

    • 项目概要计划
    • 里程碑计划
    • 项目控制计划
    • 产品控制计划
    • 验证计划
    • 运行维护计划

  • 检查进行阶段评审

  • 实现严格的产品控制

    • 实行基本配置管理

      • 基准配置又称为基线配置，它是经过阶段评审后的软件配置成分
      • 基准配置管理也称为变动控制

  • 采用现代程序设计技术

  • 结果应能清除地审查

  • 开发小组的人员应少而精

    • 人数增加，通信开销急剧增大

  • 承认不断改进软件工程实践的必要性

  • 可行性分析与项目开发计划

  • 编码

  • 测试

      • 测试用例

  • 维护

# 软件过程

  • 概念

    • 过程是活动的集合

    • 活动是任务的集合

    • 软件过程

      • 个体含义

        • 指软件产品或系统在生存周期中某一类活动的集合，如软件开发过程、软件管理过程等

      • 整体含义

        • 指软件产品或系统在所有上述含义下的软件过程的总体

      • 工程含义

        • 指解决软件过程的工程

  • 能力成熟度模型（CMM）

    • 目的

      • 提供一种评价软件承接方能力的方法

    • 级别

      • 初始级（Initial）
      • 可重复级（Repeatable）
      • 已定义级（Defined）
      • 已管理级（Managed）
      • 优化级（Optimized）

  • 能力成熟度模型集成（CMMI）

    • 是若干过程模型的综合和改进

    • 方法

      • 阶段式模型

        • 类似于 CMM

        • 成熟度等级

          • 初始的

            • 过程不可预测且缺乏控制

          • 已管理的

            • 过程为项目服务

          • 已定义的

            • 过程为组织服务

          • 定量管理的

            • 过程已度量和控制

          • 优化的

            • 集中于过程改进

      • 连续式模型

        • 关注每个过程域的能力

        • 过程域能力等级

          • CL0（未完成的）

          • CL1（已执行的）

            • 共性目标是过程将可标识的输入工作产品转换可标识的成输出工作产品

          • CL2（已管理的）

            • 共性目标集中于已管理的过程的制度化

          • CL3（已定义级的）

            • 共性目标集中于已定义的过程的制度化

          • CL4（定量管理的）

            • 共性目标集中于可定量管理的过程的制度化

          • CL5（优化的）

            • 使用量化（统计学）手段改变和优化过程域

# 软件过程模型

# 也称为软件开发模型，是软件开发全部过程、活动和任务的结构框架

# 瀑布模型（Waterfall Model）

  • 线性顺序、由前至后、相互衔接

  • 以文档为驱动、适用于软件需求很明确的项目

  • 优点

    • 容易理解
    • 管理成本低

  • 缺点

    • 对需求要求高
    • 接近结束时，才进行集成和测试
    • 项目风险控制能力弱
    • 开发费用高

# 增量模型（Incremental Model）

  • 将需求分段为一系列增量产品，每一增量可以分别开发，是交错的线性序列

  • 强调每一个增量均发布一个可操作的产品

  • 优点

    • 瀑布模型的所有优点
    • 第一个版本的开发成本和时间少
    • 可以减少用户需求的变更
    • 开发所承担的风险不大

  • 缺点

    • 初始增量导致后续增量不稳定
    • 管理成本、进度和配置复杂

# 演化模型（Evolutionary Model）

  • 专门应对不断演变的软件产品的过程模型

  • 演化模型是迭代的过程模型，适用于对软件需求缺乏准确认识的情况

  • 原型模型（Prototype Model）

    • 适用于系统规模不大且不复杂时

    • 原型

      • 原型是预期系统的一个可执行版本，反映了系统性质的一个选定子集，一个原型不必满足目标软件的所有约束，目的是能快速、低成本地构建原型
      • 探索型原型
      • 实验型原型
      • 演化型原型

  • 螺旋模型（Spiral Model）

    • 将瀑布模型和演化模型结合，加入了两种模型均忽略的风险分析，将开发过程分为几个螺旋周期

    • 螺旋周期

      • 制订计划
      • 风险分析
      • 实施工程
      • 用户评估

    • 强调风险分析，适用于庞大、复杂并且具有高风险的系统

    • 优点

      • 支持用户需求的动态变化
      • 提高软件的适应能力
      • 降低软件开发的风险

    • 缺点

      • 迭代次数过多，增加开发成本，延迟提交时间

# 喷泉模型（Water Fountain Model）

  • 是一种以用户需求为动力、以对象作为驱动的模型，适用于面向对象的开发方法

  • 允许各开发活动交叉、迭代地进行

  • 优点

    • 提高开发效率
    • 节省开发时间

  • 缺点

    • 需要大量开发人员
    • 不利于项目管理
    • 要求严格管理文档

# 基于构件的开发模型

（Component=based Development Model）

  • 是指利用预先包装的构件来构造应用系统

  • 本质上是演化模型，需要以迭代方式构建软件

  • 领域工程

    • 目的

      • 构建领域模型、领域基准体系结构和可复用构件库

    • 步骤

      • 领域分析
      • 构建领域模型和领域基准体系结构
      • 表示领域的候选构件
      • 对候选构件进行可变性分析
      • 构建可复用构件并存入可复用构件库

  • 应用系统工程

    • 目的

      • 使用可复用构件组装应用系统

    • 步骤

      • 应用系统分析
      • 设计应用系统的体系结构
      • 标识应用系统所需的构件
      • 在可复用构件库中查找合适的构件
      • 对构件进行特化或单独开发

# 形式化方法模型（Formal Methods Model）

  • 建立在严格数学基础上的一种软件开发方法
  • 主要活动是生成计算机软件形式化的数学规格说明
  • 该方法的变形时净室软件工程

# 统一过程（UP）模型

  • 用例和风险驱动，以架构为中心，迭代并且增量的开发过程

    • 迭代的意思是将整个软件开发项目划分成许多个小的“袖珍项目”，每个“袖珍项目”都包含正常软件的所有元素

  • 阶段

    • 起始阶段（Inception Phase）

      • 项目的初创活动

    • 精化阶段（Elaboration Phase）

      • 需求分析和架构演进

    • 构建阶段（Construction Phase）

      • 关注系统的构建，产生实现模型

    • 移交阶段（Transition Phase）

      • 专注于软件提交，产生软件增量

  • 核心工作流

    • 需求工作流
    • 分析工作流
    • 设计工作流
    • 实现工作流
    • 测试工作流

# 敏捷方法（Agile Development）

  • 目标是通过尽可能早地、持续地对有价值地软件的交付，使客户满意

  • 极限编程（XP）

    • 轻量级、高效、低风险、柔性、可预测、科学的软件开发方式
    • 价值观
    • 原则
    • 实践
    • 行为

  • 水晶法（Crystal）

    • 每一个不同的项目都需要一套不同的策略、约定和方法论

  • 并列征求法（Scrum）

    • 迭代
    • 按需求的优先级别实现产品
    • 并行地递增实现产品

  • 自适应软件开发（ASD）

  • 敏捷统一过程（AUP）

    • 原理

      • 在大型上连续
      • 在小型上迭代

    • 采用经典的 UP 阶段性活动，每个活动中，一个团队迭代使用敏捷，将软件增量尽快交付

    • 迭代中的活动

      • 建模
      • 实现
      • 测试
      • 部署
      • 配置及项目管理
      • 环境管理

# 需求分析

# 软件需求

  • 是指用户对目标软件系统在功能、行为、性能、设计约束等方面的期望

  • 具体内容

    • 功能需求

    • 性能需求

    • 用户或人的因素

    • 环境需求

    • 界面需求

    • 文档需求

    • 数据需求

    • 资源使用需求

    • 安全保密要求

    • 可靠性要求

    • 软件成本消耗和开发进度需求

    • 其他非功能性要求

      • 开发模式
      • 质量控制标准
      • 里程碑和评审
      • 验收标准
      • 质量要求的优先级
      • 可维护性

# 需求分析原则

  • 必须能够表示和理解问题的信息域
  • 必须能够定义软件将完成的任务
  • 必须能够表示软件的行为（作为外部事件的结果）
  • 必须划分描述数据、功能和行为的模型，从而可以分层次地揭示细节
  • 分析过程应该从要素信息移向细节信息

# 需求工程

  • 需求工程是一个不断反复的需求定义、文档记录、需求演进的过程，并最终在验证的基础上冻结需求

  • 阶段

    • 需求获取

    • 需求分析和协商

    • 系统建模

      • 面向数据流方法

      • 面向数据结构方法

      • 面向对象方法

      • 分析方法

        • 面向数据流的结构化分析方法（SA）
        • 面向对象的分析方法（OOA）

    • 需求规约

      • 是分析任务的最终产物

      • 内容

        • 引言
        • 信息描述
        • 功能描述
        • 行为描述
        • 检验标准
        • 参考书目
        • 附录

    • 需求验证

      • 目的是检验需求功能的正确性、完整性和清晰性

      • 检查内容

        • 系统定义的目标是否与用户的要求一致
        • 系统需求分析阶段提供的文档资料是否齐全；文档中的描述是否完整、清晰、准确地反映了用户要求
        • 被开发项目的数据流与数据结构是否确定且充足
        • 主要功能是否已包括在规定的软件范围之内，是否都已充分说明
        • 设计的约束条件或限制是否符合实际
        • 开发的技术风险是什么
        • 是否详细地制定了检验标准，它们能否对系统定义进行确认

    • 需求管理

      • 软件需求管理是一组用于帮助项目组在项目进展的任何时候去标识、控制和跟踪需求的活动，对需求工程所有相关活动的规划和控制

      • 赋予每个需求唯一的标识符，由此可以建立跟踪表用于需求跟踪

        • 需求跟踪的目的是为了建立和维护从用户需求开始到测试之间的一致性和完整性
        • 正向跟踪
        • 逆向跟踪

# 系统设计

# 目的是为系统制定蓝图，在各种技术和实施方法中权衡利弊，精心设计，合理地使用各种资源，最终勾画出新系统地详细设计方案

# 内容

  • 新系统总体结构设计
  • 代码设计
  • 输出设计
  • 输入设计
  • 处理过程设计
  • 数据存储设计
  • 用户界面设计
  • 安全控制设计

# 设计方法

  • 面向数据流的结构化设计方法（SD）
  • 面向对象的分析方法（OOD）

# 基本任务

  • 概要设计

    • 设计软件系统总体结构

    • 数据结构设计

      • 宜使用抽象数据类型

    • 数据库设计

      • 概念设计
      • 逻辑设计
      • 物理设计

    • 编写概要设计文档

      • 概要设计说明书
      • 数据库设计说明书
      • 用户手册
      • 修订测试计划

    • 评审

  • 详细设计

    • 对每个模型进行详细的算法设计

    • 对模块内的数据结构进行设计

    • 对数据库进行物理设计，即确定数据库的物理结构

    • 其他设计

      • 代码设计
      • 输入/输出格式设计
      • 用户界面设计

    • 编写详细设计说明书

    • 评审

# 系统设计的结果是一系列的系统设计文件，这些文件是物理实现一个信息系统的重要基础

# 系统测试

# 目的

  • 希望能以最少的人力和时间发现潜在的各种错误和缺陷

# 信息系统测试

  • 软件测试
  • 硬件测试
  • 网络测试

# 基本原则

  • 应尽早并不断地进行测试
  • 测试工作应该避免由原开发软件的人或小组承担
  • 在设计测试方案时，不仅要确定输入数据，而且要根据系统的功能确定预期输出结果
  • 在测试用例时，不仅要设计有效、合理的输入条件，也要包含不合理、失效的输入条件
  • 在测试程序时，不仅要检验程序是否做了该做的事，还要检验程序是否做了不该做的事
  • 严格按照测试计划进行，避免测试的随意性
  • 妥善保存测试计划、测试用例，作为软件文档的组成部分
  • 可以重复利用测试用例

# 测试过程

  • 制定测试计划
  • 编制测试大纲
  • 产生测试设计说明文档
  • 实施测试
  • 生成测试报告

# 传统软件的测试策略

  • 软件测试策略将软件测试用例的设计方法集成到一系列经过周密计划的步骤中

  • 任何测试策略都必须包含测试计划、测试用例设计、测试执行以及结果数据的收集和评估

  • 步骤

    • 单元测试（模块测试）

      • 侧重于模块中的内部处理逻辑和数据结构，一般用白盒测试法，可以对多个模块同时进行

      • 特征

        • 模块结构
        • 局部数据结构
        • 重要的执行路径
        • 出错信息
        • 边界条件

      • 过程

        • 驱动模块

          • 相当于一个主程序，接收测试例子的数据，将这些数据送到测试模块，输出测试结果

        • 桩模块（存根模块）

          • 用来代替测试模块中所调用的子模块，其内部可以进行少量的数据处理，目的是为了检验入口，输出调用和返回的信息

    • 集成测试

      • 集成测试是构造软件体系结构的系统化技术，同时也是进行一些旨在发现与接口相关的错误的测试，其目标是利用已通过单元测试的构件建立设计中描述的程序结构

      • 方法

        • 非增量集成

          • 分别测试各个模块，再将模块组合整体测试
          • 优点：可以并行测试
          • 缺点：容易混乱，错误难以查找和定位

        • 增量集成

          • 以小增量的方式逐步进行构造和测试

          • 优点：错误容易定位，更易对接口进行彻底测试，并且可以运用系统化的测试方法

          • 策略

            • 自顶向下集成测试

              • 从主控模块（主程序）开始，沿着控制层次逐步向下

              • 集成方法

                • 深度优先
                • 宽度优先

              • 步骤

                • 主控模块用作测试驱动模块，用这些从属于主控模块的所有模块代替桩模块，对主模块进行测试
                • 依靠所选择的集成方法，每次用实际模块代替一个从属桩模块，然后再把该桩模块的直接下属模块用新的桩模块代替，与已测试过的模块或子系统组装成新的子系统
                • 在集成每个模块后都进行测试
                • 在完成每个测试集后，用实际模块代替另一个桩模块
                • 可以执行回归测试，以确保没有引入新的错误
                • 重复 2 ~ 5 的步骤

            • 自底向上集成测试

              • 从原子模块（程序结构的最底层构件）开始进行构造和测试

              • 由于构件是自底向上集成的，在处理时所需要的从属于给定层次的模块总是存在的，因此，不需要使用桩模块

              • 步骤

                • 连接底层构件以构成完成特定子功能的簇
                • 编写驱动模块（测试的控制程序）以协调测试用例的输入和输出
                • 测试簇
                • 去掉驱动模块，沿着程序结构向上逐步链接簇

            • 回归测试

              • 重新执行已经测试过的某些子集

              • 测试用例

                • 能够测试软件所有功能的具有代表性的测试样本
                • 额外测试，侧重于可能会受变更影响的软件功能
                • 侧重于已发生变更的软件构件测试

            • 冒烟测试

              • 一种常用的集成测试方法，是时间关键项目的决定性机制，让软件团队频繁地对项目进行评估

              • 活动

                • 将已经转换为代码的软件构件集成到构建中

                • 设计一系列测试以暴露影响构建正确地完成其功能的错误

                • 每天将该构建与其他构建及整个软件产品集成起来进行冒烟测试

                  • 集成方法

                    • 自顶向下
                    • 自底向上

    • 确认测试

      • 始于集成测试的结束，集中于用户可见的动作和用户可识别的系统输出

      • 确认测试准测

        • 软件确认是通过一系列表明和软件需求相符合的测试获得

      • 配置评审

        • 检查软件、文档、数据是否齐全，分类是否有序

      • α 测试和 β 测试

        • α 测试

          • 由有代表性的最终用户在开发者的场所进行

        • β测试

          • 在一个或多个最终用户场所执行
          • 变体：客户验收测试

    • 系统测试

      • 将已经确认的软件、计算机硬件、外设和网络等其他因素结合在一起，进行信息系统的各种集成测试和确认测试，目的是通过与系统的需求相比较，发现所开发的系统与用户需求不符或矛盾的地方

      • 恢复测试

      • 安全性测试

      • 压力测试

        • 变体：敏感性测试

      • 性能测试

        • 在测试过程的任何步骤都可进行

      • 部署测试

        • 又称为配置测试

# 测试方法

  • 在软件测试过程中，应该为定义软件测试模板，即将特定的测试方法和测试用例设计放在一系列的测试步骤中

  • 软件测试方法

    • 静态测试

      • 指被测试程序不在机器上运行
      • 人工检测
      • 计算机辅助静态分析的手段

    • 动态测试

      • 通过运行程序发现错误

      • 测试用例由测试输入数据和与之对应的预期输出结果组成

      • 黑盒测试法

        • 也称为功能测试，完全不考虑软件的内部结构和特性，测试软件的外部特性

        • 错误

          • 是否有错误的功能或遗漏的功能？
          • 界面是否有误？输入是否正确接收？输出是否正确？
          • 是否有数据结构或外部数据库访问数据？
          • 性能是否能够接受？
          • 是否有初始化或终止性错误？

        • 测试技术

          • 等价类划分

            • 将程序的输入域划分为若干等价类，然后从每个等价类中选取一个代表性数据作为测试用例

            • 划分

              • 有效等价类
              • 无效等价类

          • 边界值分析

          • 错误推断

          • 因果图

            • 从自然语言描述的程序规格说明中找出因（输入）果（输出或程序的改变状态），通过因果图转换为判定表
            • 这样生成的测试用例数据最少

      • 白盒测试法

        • 也称为结构测试，根据程序的内部结构和逻辑来设计测试用例，对程序的路径和过程进行测试，检查是否满足设计的需要

        • 测试技术

          • 逻辑覆盖

            • 语句覆盖

            • 判定覆盖

            • 条件覆盖

            • 判定/条件覆盖

            • 条件组合覆盖

              • 满足条件组合覆盖的测试用例一定满足判定覆盖、条件覆盖和判定/条件覆盖

            • 路径覆盖

          • 循环覆盖

          • 基本路径测试

            • 程序中的每条可执行语句至少执行一次
            • 所有条件语句的真值状态和假值状态都测试过
            • 路径测试的起点是程序控制流图，程序控制流图中的结点代表包含一个或多个无分支的语句序列，边代表控制流

        • 原则

          • 程序模块中的所有独立路径至少执行一次
          • 在所有的逻辑判断中，取“真”值和取“假”值的两种情况至少都能执行一次
          • 每个循环都应在边界条件和一般条件下各执行一次
          • 测试程序内部数据结构的有效性等

# 调试

  • 调试发生在测试之后，其任务是根据测试时所发现的错误找出原因和具体的位置，进行改正

  • 调试过程

    • 调试过程通常得到以下两种结果之一：发现问题的原因并将其改正；未能找到问题的原因。在后一种情况下，调试人员可以假设一个原因，设计一个或多个测试用例来帮助验证这个假设，重复此过程直到改正错误
    • 高度耦合的构件可能会加剧：症状与原因出现的地方相隔甚远

  • 调试方法

    • 试探法

      • 适用于结构简单的程序

    • 回溯法

      • 适用于小型程序

    • 对分查找法

    • 归纳法

      • 从测试暴露的问题除法，收集所有正确或不正确的数据，分析它们之间的关系，提出假想的错误原因，用这些数据来证明或反驳，从而查出错误所在

    • 演绎法

      • 根据测试结果，列出所有可能的错误原因
      • 用假设来解释原始的测试结果

# 软件项目管理

# 软件的项目管理是指软件生存周期中软件管理者所进行的一系列活动，其目的是在一定的时间和预设范围内有效地利用人力、资源、技术和工具，使软件系统或软件产品按原定计划和质量要求如期完成

# 涉及范围

  • 人员（Person）

    • 软件工程项目的基本要素和关键因素

    • 分类

      • 项目管理人员
      • 高级管理人员
      • 开发人员
      • 客户
      • 最终用户

  • 产品（Product）

    • 在进行项目计划之前，应该首先进行项目定义，也就是定义项目范围，其中包括建立产品的目的和范围，可选的解决方案、技术或管理的约束等

    • 软件范围

      • 项目环境
      • 信息目标
      • 功能和性能

  • 过程（Procedure）

    • 过程控制

      • 将项目分解为任务、子任务等
      • 分解准则是基于软件工程的过程

    • 过程模型

    • 过程分解

  • 项目（Projcet）

    • 进行有计划和可控制的软件项目是管理复杂性的一种方式

    • 五个部分

      • 明确目标及过程
      • 保持动力
      • 跟踪进展
      • 做出明智的决策
      • 进行事后分析

# 软件项目估算

  • 估算方法

    • 基于已经完成的类似项目进行估算

    • 基于分解技术进行估算

      • 问题分解
      • 过程分解

    • 基于经验估算模型的估算

      • IBM 估算模型
      • CoCoMo 模型
      • Putnam 模型

  • 成本估算方法

    • 自顶向下估算

      • 工作量小、速度快
      • 不清楚低级别上的技术性困难

    • 自底向上估算

      • 估算准确
      • 忽略子任务间相互联系，估算偏低

    • 差别估算方法

      • 将待开发项目与一个或多个已完成的类似项目进行比较，找出与某个相似项目的若干不同之处

    • 其他估算方法

      • 专家估算法
      • 类推估算法
      • 算式估算法

  • COCOMO 估算模型

    • 精确的、易于使用的成本估算模型

    • 类别

      • 基本 COCOMO 模型

        • E = a(L)^b D = cE^d E：工作量（人月） D：开发时间（月） L：源代码行数估计值（千行代码） a, b, c, d：常数

      • 中级 COCOMO 模型

        • E = a(L)^b*EAF EAF：工作量调节因子

      • 详细 COCOMO 模型

        • 将软件系统模型分为系统、子系统和模块三个层次

  • COCOMOII 模型

    • 层次结构的估算模型，被分为三个阶段性模型

    • 应用组装模型

    • 早期设计阶段模型

    • 体系结构阶段模型

    • 对象点

      • 一种间接的软件测量
      • 计算对象点时使用如下的数值：屏幕数；报表数；构造应用系统可能需要的构件数

    • 功能点

      • 功能点可以转换为代码行

    • 代码行

  • Putnam 估算模型

    • 动态多变量模型
    • 假设在软件开发的整个生存周期中工作量有特定的分布
    • L = Ck*E^(1/3)*td^(4/3) L：代码行数 td：开发时间（年） E：工作量（人年） Ck：技术状态常数，其值依赖于开发环境

# 进度管理

  • 进度管理的目的是确保软件在规定的时间内按期完成

  • 两种方式

    • 系统最终交付时期已定，软件开发部门必须在规定期限内完成
    • 系统最终交付日期只确定了大致的年限，最后交付日期由软件开发部门确定

  • 基本原则

    • 划分
    • 相互依赖性
    • 时间分配
    • 工作量确认
    • 确定责任
    • 明确输出结果
    • 确定里程碑

  • 进度安排

    • 图示内容

      • 各个任务的计划开始时间和完成时间
      • 各个任务的完成标志
      • 各个任务与参与工作的任务，各个任务与工作量之间的衔接情况
      • 完成各个任务所需的物理资源和数据资源

    • 描述方法

      • Gantt 图

      • 项目计划评审技术（PERT）

        • 松弛时间
        • 关键路径

# 软件项目的组织

  • 不仅要考虑软件项目的特点，还需要考虑参与人员的素质

  • 组织原则

    • 尽早落实责任
    • 减少交流接口
    • 责权均衡

  • 组织结构的模式

    • 按项目划分的模式

      • 将开发人员组织成项目组

    • 按职能划分的模式

      • 按软件过程中所反映的各种职能将项目的参与者组织成相应的专业组

    • 矩阵模式

      • 既按职能组织相应的专业组，又按项目组织项目组

  • 程序设计小组的组织方式

    • 主程序员制小组

      • 突出主程序员的领导作用
      • 通信主要体现在主程序员和程序员之间

    • 民主制小组

    • 层次式小组

      • 一名组长领导若干名高级程序员，每名高级程序员领导若干名程序员
      • 适合于具有层次结构特征的项目的开发

# 软件配置的管理

  • 软件配置管理（SCM）用于整个软件工程过程。其主要目标是标识变更、控制变更；确保变更正确地实现；报告有关变更。SCM 是一组管理整个软件生存周期中各阶段变更地活动。

  • 基线

    • 基线是软件生存周期中个开发阶段的一个特定点，它的作用是使各开发阶段的工作划分更加明确，使本来连续的工作在这些点上断开，以便检查与肯定阶段成果。
    • 基线可以作为一个检查点，在开发过程中，当采用的基线发生错误时可以知道所处的位置，返回到最近和最恰当的基线上。

  • 软件配置项

    • 软件配置项（SCI）是软件工程中产生的信息项，它是配置管理的基本单位
    • 有些 SCI 是 SCM 的对象，并可形成基线

  • 版本控制

    • 动态的概念
    • 软件的每一个版本都是 SCI 的一个汇集

  • 变更控制

    • 是一项最重要的软件配置任务

    • 需借助于配置数据库和基线的概念

    • 配置数据库类别

      • 开发库

      • 受控库

# 软件容错技术

  • 提高软件质量和可靠性的技术类别

    • 避开错误
    • 容错技术

  • 容错软件

    • 对自身错误有一定的屏蔽能力
    • 在一定程度上能从错误状态自动恢复到正常状态
    • 在发生错误时仍能在一定程度上完成预期的功能
    • 在一定程度上具有容错能力

  • 容错的一般方法

    • 冗余

      • 冗余是指对于实现系统规定功能是多余的那部分资源，包括硬件、软件、信息和时间。由于加入了这些资源，有可能使系统的可靠性得到较大的提高

      • 冗余技术

        • 结构冗余

          • 静态冗余

            • 三模冗余（TMR）
            • 多模冗余

          • 动态冗余

            • 多重模块待机储备

            • 热备份系统

              • 备用模块和主模块一同工作
              • 备用模块在待机过程中的失效率为 0

            • 冷备份系统

              • 备用模块不工作

          • 混合冗余

        • 信息冗余

          • 奇偶码
          • 循环码

        • 时间冗余

          • 以重复执行指令或程序来消除瞬时错误带来的影响

          • 处理办法

            • 中断
            • 复算
            • 重新组合系统
            • 放弃程序处理

        • 冗余附加技术

          • 冗余附加技术是指为实现上述冗余技术所需的资源和技术，包括程序、指令、数据、存放和调动它们的空间和通道等

          • 屏蔽硬件错误

            • 关键程序和数据冗余存储及调用
            • 检测、表决、切换、重构、纠错和复算的实现

          • 屏蔽软件错误

            • 冗余备份程序的存储及调用
            • 实现错误检测和错误恢复的程序
            • 实现容错软件所需的固化程序

# 软件度量

# 软件度量用于对产品及开发产品的过程进行度量。软件产品、软件过程、资源都具有外部属性和内部属性。

  • 外部属性

    • 面向管理者和用户
    • 体现了软件产品/软件过程与相关资源和环境的关系
    • 经常可采用直接测量的方法进行

  • 内部属性

    • 指软件产品或软件过程本身的属性
    • 只能用间接测量的方法度量
    • 需要一定的测量方法或模型

# 软件度量分类

  • 第一种分类方法

    • 面向规模的度量

      • 用于收集与软件规模相关的软件工程输出信息和质量信息

      • 是通过对质量和（或）生产率的测量进行规范化得到的，而这些量都是根据开发过的软件的规模得到的

        • 软件规模通常用程序的代码行（LOC）或千行代码（KLOC）来衡量

        • 面向规模的度量公式

          • LOC 或 KLOC

            • 代码行数或千行代码数

          • 生产率 P

            • P = LOC / E E 为开发的工作量（人月）

          • 每行代码平均成本 C

            • C = S / LOC S 为总成本

          • 文档代码比 D

            • D = Pe / KLOC Pe 为文档页数

          • 代码错误率 EQR

            • EQR = N / KLOC N 为代码中的错误数

    • 面向功能的度量

      • 集中在软件的“功能性”和“实用性”

      • 以功能测量数据作为规范化值

      • 功能点

        • 功能点是根据软件信息域的特性及复杂性来计算的

        • 功能点计算

          • Fp = 总计 × [0.65 + 0.01 × Σ(Fi)] 总计：所有 Fp 项的总数 Fi(i = 1 ~ 14) 是值调整因子

      • 信息域的值

        • 外部输入数（EI）

          • 输入常用于更新内部逻辑文件 ILF
          • 输入应该与独立计数的查询区分开来

        • 外部输出数（EO）

          • 不对报告中的单独数据项进行分开计数

        • 外部查询数（EQ）

          • 一个外部查询定义为一个在线输入
          • 其结果是以在线输出（经常从 ILF 中得到）的方式产生某个即时软件响应

        • 内部逻辑文件数（ILF）

          • 每个内部逻辑文件是驻留在应用系统【边界之内】的数据逻辑分组，它通过【外部输入】来维护

        • 外部接口文件数（EIF）

          • 每个外部接口文件是驻留在应用系统【外部】的数据逻辑分组，它通过应用系统提供有用的信息

    • 面向人的度量

      • 收集有关人们开发软件所用方式的信息和人员理解有关工具的方法和效率的信息
      • 还有基于问题、基于过程、基于用例等成本估算方法

  • 第二种分类方法

    • 生产率度量

      • 关注于软件工程活动的制品

    • 质量度量

      • 可指明软件满足明确的和隐含的用户需求的程度

    • 技术度量

      • 集中在软件产品的某些特征

# 软件复杂性度量

  • 软件复杂性是指理解和处理软件的难易程度

  • 参数

    • 规模

      • 指令数或源程序行数

    • 难度

      • 操作数的数目

    • 结构

      • 程序结构有关的度量

    • 智能度

      • 算法的难易程度

  • 分类

    • 程序复杂性

      • 描述

        • 程序理解的难度
        • 纠错、维护程序的难度
        • 向他人解释程序的难度
        • 根据设计文件编写程序的工作量
        • 执行程序时需要资源的程度

      • 原则

        • 程序复杂性与程序大小的关系不是线性的
        • 控制结构复杂的程序较复杂
        • 数据结构复杂的程序较复杂
        • 转向语句使用不当的程序较复杂
        • 循环结构比选择结构复杂，选择结构比顺序结构复杂
        • 语句、数据、子程序和模块在程序中的次序对复杂性有影响
        • 全局变量、非局部变量较多时程序较复杂
        • 函数的隐式副作用相对于显式参数传递而言更加难以理解
        • 具有不同作用的变量共用一个名字时较难理解
        • 模块间、过程间联系密切的程序比较复杂
        • 嵌套程度越深，程序越复杂

      • 程序复杂性度量

        • McCabe 环路复杂性度量

          • 基于程序控制流的复杂性度量方法

          • 认为程序的复杂性很大程度上取决于控制的复杂性

          • 又称为环路度量

          • 程序图

            • 退化的程序流程图
            • 把程序流程图中的每个处理符号都退化成一个结点
            • 原来连接不同处理符号的流线变成连接不同点的有向弧
            • 仅描述程序内部的控制指令，完全不表现对数据的具体操作以及分支和循环的具体条件

          • 用程序图的环路数作为程序复杂性的度量值

            • V(G) = m - n + 2p G：有向图 V(G)：环路个数 m：弧的个数 n：结点个数 p：强连通分量个数
            • 在一个程序中，从程序图的入口点总能到达图中的任何一个结点，因此，程序总是连通的，但不是强连通的
            • 为了使程序成为强连通图，从图的入口点到出口点加一条用虚线表示的有向边，使图成为强连通图
            • 当模块的环路复杂度超过 10 时，要充分测试这个模块会变得特别难

        • Halstead 复杂性度量

    • 文档复杂性

# 计算机辅助软件工程（CASE）

（Computer Aided Software Engineering）

  • 是指使用计算机及相关的软件工具辅助软件开发、维护、管理等过程中各项活动的实施，以确保这些活动能高效率、高质量地进行

# 软件工具

  • 用来辅助软件开发、运行、维护、管理和支持等过程中的活动的软件称为软件工具

  • 软件开发工具

    • 需求分析工具

      • 辅助软件需求分析活动的软件

      • 辅助系统分析员从需求定义出发，生成完整的、清晰的、一致的功能规范（Functional Specification）

        • 功能规范是指软件所要完成功能的准确而完整的陈述

      • 分类

        • 基于自然语言或图形描述的工具
        • 基于形式化需求定义语言的工具

    • 设计工具

      • 辅助软件设计活动的软件

      • 辅助设计人员从软件功能规范出发，得到相应的设计规范（Design Specification），对应于概要设计活动和详细设计活动

      • 分类

        • 概要设计工具

          • 辅助设计人员设计目标软件的体系结构、控制结构和数据结构

        • 详细设计工具

          • 辅助设计人员设计模块的算法和内部实现细节

        • 基于形式化描述的设计工具

        • 面向对象的分析与设计工具

    • 编码与排错工具

      • 辅助程序员进行编码活动的工具

    • 测试工具

      • 支持软件测试的工具

      • 分类

        • 数据获取工具

        • 静态分析工具

          • 对源程序的程序结构、数据流和控制流进行分析
          • 发现语义错误

        • 动态分析工具

          • 通过执行程序检查语句、分支和路径覆盖，测试有关变量值的断点，即对程序的执行流进行探测

        • 模拟工具

        • 测试管理工具

  • 软件维护工具

    • 辅助软件维护过程中活动的软件称为软件维护工具，它辅助维护人员对软件代码及其文档进行各种维护活动

    • 分类

      • 版本控制工具

      • 文档分析工具

        • 用来对软件开发过程中形成的文档进行分析，给出软件维护活动所需的维护信息

      • 开发信息库工具

        • 用来维护软件项目的开发信息，包括对象、模块等

      • 逆向工程工具

        • 辅助软件人员将某种形式表示的软件转换成更高抽象形式表示的软件

        • 力求恢复源程序的设计信息，使软件变得更容易理解

        • 分类

          • 静态
          • 动态

      • 再工程工具

        • 用来支持重构一个功能和性能更为完善的软件系统

        • 集中点

          • 代码重构
          • 程序结构重构
          • 数据结构重构

  • 软件管理和软件支持工具

    • 用来辅助管理人员和软件支持人员的管理活动和支持活动，以确保软件高质量地完成

    • 分类

      • 项目管理工具

        • 用来辅助软件的项目管理活动

      • 配置管理工具

        • 辅助完成软件配置项的标识、版本控制、变化控制、审计和状态统计等基本任务

      • 软件评价工具

        • 用来辅助管理人员进行软件质量保证的有关活动
        • 有助于软件质量控制

# 软件开发环境

（Software Development Environment）

  • 指支持软件产品开发的软件系统，它由软件工具集和环境集成机制构成

  • 工具集

    • 用于支持软件开发的相关过程、活动和任务

  • 环境集成机制

    • 为工具集成和软件开发、维护和管理提供统一的支持

    • 分类

      • 数据集成

        • 为各种相互协作的工具提供统一的数据模式和数据接口规范，以实现不同工具之间的数据交换

      • 界面集成

        • 指环境中的工具的界面使用统一的风格，采用相同的交互方法，提供一种相似的视感效果
        • 可以减少用户学习不同工具的开销

      • 控制集成

        • 用于支持环境中各个工具或开发活动之间的通信、切换、调度和协同工作，并支持软件开发过程的描述、执行与转换

      • 其他方面的集成

        • 平台集成

          • 指在不同的硬件和系统软件之上构造用户界面一致的开发平台，并集成到统一的环境中

        • 方法与过程集成

          • 指把多种开发方法、过程模型及其相关工具集成在一起

  • 特征

    • 环境的服务是集成的

      • 软件开发环境应支持多种集成机制

    • 环境应支持小组工作方式，并为其提供配置管理

    • 环境的服务可用于支持各种软件开发活动

  • 集成型开发环境

    • 是一种把支持多种软件开发方法和开发模型的软件工具集成在一起的软件开发环境

    • 特点

      • 开放性

        • 为环境外的工具集成到环境中提供方便

      • 可剪裁性

        • 可根据不同的应用和不同的用户需求进行剪裁，以形成特定的开发环境
