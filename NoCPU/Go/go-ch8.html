# Goroutines 和 Channels

  • Go 中的并发可以用两种手段来实现，本章中的 goroutine 和 channel 支持**“顺序通信进程”（CSP）**，在这种编程模型中值会在不同的运行实例（goroutine）中传递，尽管大多数情况下仍然是被限制在单一实例中。

  • Goroutines：每一个并发单元叫做一个 goroutine。当一个程序启动时，其主函数在一个单独的 goroutine 中运行（main goroutine）。通过 go 语句来创建新的 goroutine，go 语句会使其与语句中的函数在一个新创建的 goroutine 中运行，而 go 语句本身会快速地完成。

    f() // 需要等待 f 执行完成
go f() // 创建新的 goroutine，不需要等待 f
    1
    2

    例子：当主函数返回时，所有 goroutine 都会被打断。除了从主函数退出或者直接终止程序之外，没有其他的编程方法能够让一个 goroutine 来打断另一个的执行，但可以通过 goroutine 之间的通信来让一个 goroutine 请求其他的 goroutine，并让被请求的 goroutine 自行结束执行。

    // spinner 和 fib 这两个函数会同时执行
func main() {
	go spinner(100 * time.Millisecond)
	const n = 45
	fibN := fib(n)
	fmt.Printf("\rFibonacci(%d) = %d\n", n, fibN)
}

// 动效函数，当 fib 递归正在执行时，这个函数也在执行
// 当主函数返回时，所有 goroutine 都会被打断
func spinner(delay time.Duration) {
	for {
		for _, r := range `-\|/` {
			fmt.Printf("\r%c", r)
			time.Sleep(delay)
		}
	}
}

func fib(x int) int {
	if x < 2 {
		return x
	}
	return fib(x-1) + fib(x-2)
}

  • 让服务器并发不只是处理多个客户端的请求时需要，甚至在处理单个连接的多个请求时也会用到。然而在使用 go 关键词的同时，需要慎重地考虑 net.Conn 中的方法在并发地调用时是否安全。

  • Channels：是 goroutine 之间的通信机制，每个 channel 都有一个特殊的类型（可发送数据的类型）。

    ch := make(chan int, 0) // 第二个参数对应 channel 的容量
    1

    • channel 也对应一个 make 创建的底层数据结构的引用。当我们复制一个 channel 或用于函数参数传递时，我们只是拷贝了一个 channel 引用。channel 的零值时 nil。

    • 两个相同类型的 channel 可以用 == 比较，如果引用相同的对象，则比较结果为真；channel 也可以和 nil 进行比较。

    • 通信行为：

      • 发送：ch <- x
      • 接收：x = <-ch <-ch // 省略结果的接收

    • 关闭 channel：关闭后基于该 channel 的任何发送操作都将导致 panic 异常。对一个已经被关闭的 channel 进行接收操作仍然可以接收到之前已经成功发送过的数据；如果 channel 中已经没有数据的话将产生一个零值的数据。

      close(ch)
      1

  • 不带缓存的 Channels（同步 Channels）：一个基于无缓存 Channels 的发送操作将导致发送者 goroutine 阻塞，直到另一个 goroutine 在相同的 Channels 上执行接收操作，当发送的值通过 Channels 成功传输之后，两个 goroutine 才可以继续执行之后的语句。反之，接收操作先发生，则接收者 goroutine 也会阻塞，直到有另一个 goroutine 在相同的 Channels 上执行发送操作。

    当通过一个无缓存 Channels 发送数据时，接收者收到数据发生在再次唤醒发送者 goroutine 之前

    通过 channel，当用户关闭标准输入时，主 goroutine 等待后台 goroutine 接收服务器数据结束后再退出：

    func main() {
	conn, err := net.Dial("tcp", "localhost:8000")
	if err != nil {
		log.Fatal(err)
	}
	done := make(chan struct{})
	go func() {
		io.Copy(os.Stdout, conn)
		log.Println("done")
		done <- struct{}{}
	}()
	mustCopy(conn, os.Stdin)
	conn.Close()
  <-done // 如果没有的话，os.Stdin 关闭后，main 直接退出，可能第 8 行还没执行完
}

func mustCopy(dst io.Writer, src io.Reader) {
	if _, err := io.Copy(dst, src); err != nil {
		log.Fatal(err)
	}
}

    当用户关闭了标准输入，主goroutine中的mustCopy函数调用将返回，然后调用conn.Close()关闭读和写方向的网络连接。关闭网络连接中的写方向的连接将导致server程序收到一个文件（end-of-file）结束的信号。关闭网络连接中读方向的连接将导致后台goroutine的io.Copy函数调用返回一个“read from closed connection”（“从关闭的连接读”）类似的错误，因此我们临时移除了错误日志语句。

    基于 Channels 发送消息有两个重要方面。首先每个消息都有一个值，但是有时通讯的事实和发生的时刻也同样重要。当我们更希望强调通讯发生的时刻时，我们将它称为消息事件。有些消息事件并不携带额外的信息，它仅仅是用作两个 goroutine 之间的同步，这时候可以用空结构体作为 channels 元素的类型，当然也可以使用 int 或 bool。

  • 串联的 Channels（Pipeline）：Channels 也可以用于将多个 goroutine 连接在一起，一个 Channel 的输出作为下一个 Channel 的输入（管道）。

    func main() {
	naturals := make(chan int)
	squares := make(chan int)

	go func() {
		for x := 0; ; x++ {
			naturals <- x
		}
	}()

	go func() {
		for {
			x := <-naturals
			squares <- x * x
		}
	}()

	for {
		fmt.Println(<-squares)
	}
}

    让 Channels 只发送有限的数据：没有办法直接测试一个 channel 是否已经被关闭

    func main() {
	naturals := make(chan int)
	squares := make(chan int)

	go func() {
		for x := 0; x < 100; x++ {
			naturals <- x
		}
		close(naturals)
	}()

	go func() {
		for {
			x, ok := <-naturals
			if !ok { // 必须判断，不然接收已关闭的 Channel 仍然会收到零值
				break
			}
			squares <- x * x
		}
		close(squares)
	}()

	for {
		x, ok := <-squares
		if !ok {
			break
		}
		fmt.Println(x)
	}
}

    在一个更复杂的程序中，可以通过 defer 语句关闭对应的 channel。

  • 单方向的 Channel：例如上面例子中的第二个 goroutine，它只接收 naturals，只发送 squares，即使用方式是固定单向的。尤其当一个 channel 作为一个函数参数时，它一般总是被专门用于只发送或者只接收。

    Go 提供了单方向的 channel 类型，分别用于只发送或只接收。chan<- int 表示只发送 int 的 channel，<-chan int 表示只接收 int 的channel。这种限制将在编译期检测。

    因为关闭操作只用于断言不再向 channel 发送新的数据，所以只有在发送者所在的 goroutine 才会调用 close 函数，对一个只接收的 channel 调用 close 将是编译错误。

    func counter(out chan<- int) {
	for x := 0; x < 100; x++ {
		out <- x
	}
	close(out)
}

func squarer(out chan<- int, in <-chan int) {
	for v := range in {
		out <- v * v
	}
	close(out)
}

func printer(in <-chan int) {
	for v := range in {
		fmt.Println(v)
	}
}

func main() {
	naturals := make(chan int)
	squares := make(chan int)
  // 函数传参时会进行隐式类型转换
	go counter(naturals) // chan int => chan<- int
	go squarer(squares, naturals)
	printer(squares)
}

  • 带缓存的 Channels：内部持有一个元素队列，队列的最大容量是在调用 make 函数创建 channel 时通过第二个参数指定的。向缓存 Channel 发送操作就是向缓存队列的尾部插入元素，接收则是从头部删除元素。如果内部缓存队列是满的，那么发送操作将阻塞直到因另一个 goroutine 执行接收操作而释放了新的队列空间；如果 channel 是空的，接收操作将阻塞直到有另一个 goroutine 执行发送操作而向队列插入元素；当队列既不满也不空时，接收和发送操作都不会发生阻塞。

    channel 内部缓存的容量可以通过 cap(ch) 来查询；channel 内部缓存队列中有效元素的个数可以通过 len(ch) 来查询（但在并发程序中该信息会随着其他 goroutine 的操作而失效）。

    func mirroredQuery() string {
    responses := make(chan string, 3)
    go func() { responses <- request("asia.gopl.io") }()
    go func() { responses <- request("europe.gopl.io") }()
    go func() { responses <- request("americas.gopl.io") }()
    return <-responses // return the quickest response
}
// mirroredQuery 函数可能在另外两个响应慢的站点响应之前就返回了结果

func request(hostname string) (response string) { /* ... */ }

    如果使用无缓存的 channel，那么两个慢的 goroutines 将会因为没有人接收而被永远卡住。这种情况，称为 goroutines 泄漏，不会被自动回收，因此确保每个不再需要的 goroutines 能正常退出是重要的。

    无缓存 Channel 更强地保证了每个发送操作和相应的接收操作间的同步性；带缓存 Channel 对这些操作是解耦的。

    即使我们知道将要发送到一个 channel 的信息的数量上限，创建一个对应容量大小的带缓存 channel 也是不现实的，因为这要求在执行任何接收操作之前缓存所有已经发送的值，如果未能分配足够的缓存将导致程序死锁。

      这个版本的函数又有一个 bug，就是当遇到一个非 nil 的 error 时，它会直接 return，使得之后的 channel 无法被排空，这样剩下的 worker goroutine 在向它发送信息时，会被永远地阻塞，永远不会退出。（goroutinue 泄漏）

    5. 解决办法是采用一个合适大小的 buffered channel（或者创建另外一个 goroutine，当 main goroutine 返回第一个错误的同时去排空 channel）

      func makeThumbnails5(filenames []string) (thumbfiles []string, err error) {
	type item struct {
		thumbfile string
		err error
	}

	ch := make(chan item, len(filenames))
	for _, f := range filenames {
		go func(f string) {
			var it item
			it.thumbfile, it.err = thumbnail.ImageFile(f)
			ch <- it
		}(f)
	}

	for range filenames {
		it := <-ch
		if it.err != nil {
			return nil, it.err
		}
		thumbfiles = append(thumbfiles, it.thumbfile)
	}

	return thumbfiles, nil
}

    6. 最后一个版本返回新文件们的大小总计数，并且将文件名通过 channel 传递过来（而不是 slice），所以无法对循环次数进行预测。为了知道最后一个 goroutine 什么时候结束，需要一个递增的计数器，在每个 goroutine 启动时 + 1，在 goroutine 退出时 - 1，并且需要安全地改变，在减为 0 时一直等待。这种计数类型被称为 sync.WaitGroup

      func makeThumbnails6(filenames <-chan string) int64 {
	sizes := make(chan int64)
	var wg sync.WaitGroup
	for f := range filenames {
		wg.Add(1)
    // worker
		go func(f string) {
			defer wg.Done()
			thumb, err := thumbnail.ImageFile(f)
			if err != nil {
				log.Println(err)
				return
			}
			info, _ := os.Stat(thumb)
			sizes <- info.Size()
		}(f)
	}

  // closer
	go func() {
		wg.Wait()
		close(sizes)
	}()

	var total int64
	for size := range sizes {
		total += size
	}

	return total
}

      注意 Add 和 Done 方法的不对称。Add 是为计数器加一，必须在 worker goroutine 开始之前调用，而不是在 goroutine 中；否则的话我们没办法确定 Add 是在 "closer" goroutine 调用 Wait 之前被调用。并且 Add 还有一个参数，但 Done 却没有任何参数；其实它和 Add(-1) 是等价的。我们使用 defer 来确保计数器即使是在出错的情况下依然能够正确地被减掉。

      closer goroutine 和 sizes loop goroutine 必须是并发的，如果 Wait 就放在函数内部、在 sizes loop 之前，就会永远等待（因为如果没有读取 sizes channel，wg.Done 就无法执行）；如果在 sizes loop 之后，loop 就无法终止（因为没有 closer 去 close(sizes)）。

  • 并发的退出：由于一般不清楚同一时间有多少 goroutine 在运行，需要一个广播机制，让 goroutine 们知道应该结束退出了。当关闭一个 channel，并且消费掉其所有已发送的值后，再接收 channel 会立即产生零值，所以可以通过关闭一个 channel 来进行广播。

    我们创建一个退出的 channel，不向它发送任何值，但其所在的闭包内要写明程序需要退出。同时定义一个函数来轮询退出状态。也就是说，如果该 channel 可以无阻塞被读取到了，那么说明该 channel 已经被关闭了。

    func walkDir(dir string, n *sync.WaitGroup, fileSizes chan<- int64) {
	defer n.Done()
	// 这样可以将所有在取消事件之后创建的 goroutine 改变为无操作
	if cancelled() {
		return
	}

			subdir := filepath.Join(dir, entry.Name())
			go walkDir(subdir, n, fileSizes)

		}

    type client chan<- string

var (
	entering = make(chan client)
	leaving  = make(chan client)
	messages = make(chan string)
)

func boardcaster() {
	clients := make(map[client]bool)
	for {
		select {
		case msg := <-messages:
			// 广播信息给所有在线的客户端
			for cli := range clients {
				cli <- msg
			}
		case cli := <-entering:
			clients[cli] = true
		case cli := <-leaving:
			delete(clients, cli)
			close(cli)
		}
	}
}

func clientWriter(conn net.Conn, ch <-chan string) {
	for msg := range ch {
		fmt.Fprintln(conn, msg)
	}
}

func handleConn(conn net.Conn) {
	ch := make(chan string)
	go clientWriter(conn, ch)

	who := conn.RemoteAddr().String()
	ch <- "You are " + who // 发给该客户端
	entering <- ch
	messages <- who + " has arrived"

	input := bufio.NewScanner(conn)
	for input.Scan() {
		messages <- who + ": " + input.Text()
	}

	leaving <- ch
	messages <- who + " has left"
	conn.Close()
}

func main() {
	listener, err := net.Listen("tcp", "localhost:8000")
	if err != nil {
		log.Fatal(err)
	}
	go boardcaster()
	for {
		conn, err := listener.Accept()
		if err != nil {
			log.Print(err)
			continue
		}
		go handleConn(conn)
	}
}
